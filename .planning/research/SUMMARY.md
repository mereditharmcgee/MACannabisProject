# Project Research Summary

**Project:** MA Cannabis Dispensary Ownership Directory
**Domain:** SEO-optimized static directory site with client-side search (~525 records)
**Researched:** 2026-03-17
**Confidence:** HIGH

## Executive Summary

This project is a search-first, SEO-optimized static directory site that answers one question no existing platform answers: who owns each Massachusetts cannabis dispensary? The recommended approach is a fully static site generated by Astro 5, deployed to Cloudflare Pages, with client-side fuzzy search via Fuse.js. All 525+ dispensary pages are pre-rendered at build time from a spreadsheet-to-JSON pipeline, making the site fast, SEO-friendly, and trivially deployable. The architecture is well-understood, the tooling is mature, and the dataset is small enough that every "should we add complexity?" question answers itself: no.

The site's unique value — ownership tags, MSO vs. independent classification, and ownership narratives — fills a genuine competitive gap that Leafly, Weedmaps, and even the Massachusetts Cannabis Control Commission do not address. The recommended approach doubles down on this by optimizing every page for "who owns [dispensary name]" long-tail queries using LocalBusiness-adjacent schema, per-page unique narratives, and cross-linking by owner. These differentiators are low-complexity but high-value, and they are already in the dataset.

The dominant risks are not technical. Thin content penalties from Google's 2025 spam update are the biggest threat — the 525 templated pages must have genuinely unique ownership narratives, not formulaic one-liners. Inaccurate ownership data creates legal exposure since this is first-party editorial content without Section 230 protection. Both risks must be addressed before launch: unique narrative content in the data pipeline and a "Suggest a correction" form with disclaimers shipped as v1 requirements, not afterthoughts.

## Key Findings

### Recommended Stack

Astro 5.17 with Tailwind CSS 4.2, Fuse.js 7.1, and TypeScript 5.x forms the core stack. Astro is purpose-built for this use case: Content Collections with the `file()` loader ingests the CSV natively, and `getStaticPaths()` generates one pre-rendered HTML page per dispensary at build time. Cloudflare Pages is the deployment target — free tier, git-triggered builds, automatic edge CDN. The data pipeline is a simple Node.js script using PapaParse to convert the spreadsheet CSV to validated JSON. Deliberately avoid Astro 6 (released March 10, 2026 — only 7 days old at research time; requires Node 22 and Zod 4, no battle-tested ecosystem yet).

**Core technologies:**
- **Astro 5.17:** SSG framework — purpose-built for content-heavy static sites with built-in Content Collections for spreadsheet-to-pages pipeline
- **Tailwind CSS 4.2:** Utility CSS — CSS-first config, 5x faster builds, ideal for card grids and filter pill UI
- **Fuse.js 7.1:** Client-side fuzzy search — 6KB, zero dependencies, structured field search over JSON (vs. Pagefind's full-text index); 525 records trivially fit in browser memory
- **TypeScript 5.x:** Type safety — catches schema drift when spreadsheet is updated; Astro handles compilation
- **PapaParse 5.x:** CSV parsing — handles Excel export edge cases; used in Node build script only, not shipped to browser
- **@astrojs/sitemap:** XML sitemap — generates sitemap with all 525+ pages for Google indexing

### Expected Features

All table-stakes features align with what the dataset already contains — they require zero new data collection. Differentiators (ownership tags, MSO toggle, cross-linking) are also already in the data, making the feature set unusually achievable for MVP.

**Must have (table stakes):**
- Instant search with typeahead — users will not browse 525 listings manually
- Mobile-responsive card grid — 60%+ of directory searches are mobile
- Individual detail pages for all 525 dispensaries — core SEO landing targets
- County/town location filter — basic geographic narrowing
- NAP (name, address, phone) data on each listing — basic directory expectation
- Clear ownership information prominently displayed — the site's entire reason for existing

**Should have (differentiators):**
- Ownership characteristic badge filters (Women-Owned, Black-Owned, Veteran-Owned, Social Equity, LGBTQ+-Owned, Independent) — nobody else offers this
- MSO vs. Independent toggle — unique filter, trivial to implement, powerful for values-driven consumers
- Ownership narratives (2-3 sentences per listing) — unique content Google values and competitors cannot replicate
- Cross-linking by owner/parent company — connects the ownership graph no other site maps
- "Suggest a correction" form — trust signal and legal defense; must ship with v1
- Schema.org structured data on every detail page — rich snippet opportunities

**Defer (v2+):**
- Map view — addresses "what's near me?" not "who owns it?"; defer until analytics show geographic browsing demand
- Owner claim/edit flow — requires auth, moderation, abuse prevention; only build if dispensary owners request it
- Multi-state expansion — validate MA model before scaling
- Real-time MCC data integration — manual monthly updates are sustainable and preserve the researched ownership context

### Architecture Approach

The architecture is a clean three-layer system: (1) build-time data pipeline (CSV to validated JSON via Astro Content Collections), (2) Astro SSG generating 525+ pre-rendered HTML detail pages plus owner group pages, and (3) a minimal client-side search island (vanilla TypeScript + Fuse.js initialized from a pre-built index). The search JSON index is computed at build time using `Fuse.createIndex()` and served as static files (~70KB total), so browser initialization is a fast JSON parse rather than expensive index computation. Detail pages ship zero JavaScript — they are fully rendered HTML with JSON-LD structured data that crawlers read directly.

**Major components:**
1. **Data Pipeline** — CSV parsing, Zod schema validation, JSON output; blocks everything else; must fail the build on invalid data
2. **Page Generator (Astro SSG)** — `getStaticPaths()` produces 525 dispensary pages + owner group pages; all pre-rendered with structured data
3. **Search Index Builder** — build-time script using `Fuse.createIndex()`; outputs `search-data.json` (~40KB) and `search-index.json` (~30KB) to `public/`
4. **Search + Filter UI (client-side island)** — vanilla TypeScript; Fuse.js fuzzy search + attribute filters operating on the same in-memory dataset; no server round-trips
5. **Cloudflare Pages CDN** — static hosting, git-triggered builds, automatic edge caching; no platform lock-in

### Critical Pitfalls

1. **Thin content penalty (Google's August 2025 spam update)** — 525 templated pages with formulaic ownership descriptions get deindexed. Prevention: each listing must have a genuinely unique 2-3 sentence narrative with specific details (founding story, equity program context, number of locations). This is a content problem that must be solved in the data pipeline, before first build.

2. **Spreadsheet data pipeline breaks silently** — inconsistent capitalization, missing fields, encoding issues, and column renames produce broken pages that deploy without error. Prevention: a validation script that runs before `astro build` and fails the build on invalid data. Non-negotiable — validation must exist before the first deployment.

3. **Inaccurate ownership data creates legal exposure** — first-party editorial content has full publisher liability (no Section 230 protection). Prevention: "Suggest a correction" form on every listing, prominent "Last verified" date, source attribution, and Terms of Service page must all ship with v1.

4. **Incorrect schema.org markup confuses Google** — using `LocalBusiness` schema on a directory page (you are not the business) can cause Knowledge Panel confusion. Prevention: use `WebPage` + `ItemPage` with `LocalBusiness` as the `about` property; validate all 525 pages with Google's Rich Results Test before launch.

5. **Search feels broken on mobile before index loads** — cold-start delay on 3G/4G causes users to assume the site is broken and leave. Prevention: show a loading state; ensure the card grid and filter pills render on first paint without JavaScript so users can browse while the index loads.

## Implications for Roadmap

Based on research, the architecture's explicit build-order dependency chain maps directly to a natural phase structure. ARCHITECTURE.md already defines this order; the roadmap should follow it precisely.

### Phase 1: Data Foundation
**Rationale:** Every other phase depends on clean, validated data. Nothing can be built without a working pipeline. This is also where the biggest pitfall (thin content) must be addressed — in the data, not in the templates.
**Delivers:** Validated JSON dataset from spreadsheet, Zod schema, build-time validation script, canonical owner/company name normalization, "last verified" dates and source fields per record
**Addresses:** Instant search, individual detail pages, all ownership features (all depend on data)
**Avoids:** Spreadsheet pipeline breakage (Pitfall 4), thin content penalty (Pitfall 1 — unique narrative field enforced in schema)

### Phase 2: Core Static Site (Base Layout + Detail Pages)
**Rationale:** Pre-rendered detail pages deliver SEO value independently of search. Google can index all 525 pages the moment they deploy, even before the search UI exists. Establishes the URL structure and page templates that all future components reference.
**Delivers:** Astro project scaffold, BaseLayout, 525 `/dispensary/[slug]/` pre-rendered HTML pages with ownership info, address, and schema.org structured data; owner group pages `/owner/[slug]/`; sitemap.xml
**Uses:** Astro 5.17, Tailwind CSS 4.2, @astrojs/sitemap, astro-seo, schema-dts
**Avoids:** Age gate blocking crawlers (Pitfall 2 — no gate, fully public), incorrect structured data (Pitfall 6 — designed into templates)

### Phase 3: Search System
**Rationale:** Search is the primary interaction model for the homepage but depends on stable data schemas and page slugs from Phase 2. Building search after pages ensures slug consistency between search results and detail page URLs.
**Delivers:** Build-time search index generation (`Fuse.createIndex()`), `search-data.json` + `search-index.json` static assets, search-first homepage with instant typeahead, loading states, browse fallback (card grid visible before index loads)
**Uses:** Fuse.js 7.1, vanilla TypeScript search island
**Avoids:** Client-side search broken on mobile (Pitfall 5 — browse fallback ships simultaneously with search)

### Phase 4: Filters and Cross-Linking
**Rationale:** Filters build on top of the search infrastructure from Phase 3. They operate on the same in-memory dataset. Cross-links (owner pages to dispensary pages and back) depend on owner group pages from Phase 2 and stable slugs.
**Delivers:** Ownership badge filter pills (Women-Owned, Black-Owned, Veteran-Owned, Social Equity, LGBTQ+-Owned), MSO vs. Independent toggle, county/town location filter, filter state encoded in URL query parameters, cross-links on detail pages ("Other locations by [owner]")
**Implements:** Filter System and Cross-linking architecture components

### Phase 5: Trust, Polish, and Launch Readiness
**Rationale:** Legal and trust features (correction form, disclaimers, ToS) must ship with v1 — they are not polish, they are risk mitigation. SEO metadata polish and the "looks done but isn't" checklist complete the launch package.
**Delivers:** "Suggest a correction" form on every detail page, site-wide data disclaimer with "Last verified" dates, Terms of Service page, branded 404 page, Open Graph meta tags, canonical tags on all pages, robots.txt verification, accessibility audit (keyboard nav, WCAG AA contrast), empty filter state handling, favicon
**Avoids:** Inaccurate data legal exposure (Pitfall 3), missing canonical/sitemap/meta (PITFALLS.md checklist)

### Phase Ordering Rationale

- **Data before everything:** ARCHITECTURE.md is explicit — the data pipeline blocks all other components. Attempting to build pages or search without validated data produces a brittle foundation.
- **Pages before search:** SEO value accrues the moment detail pages are deployed. Search is high-value but secondary to Google indexing the 525 individual pages. This also ensures URL slugs are stable before the search index references them.
- **Search before filters:** Filters operate on the same dataset and DOM as search results. Building the search UI first creates the card grid that filters act on.
- **Trust features in the final phase but not as optional:** Pitfall 3 (legal exposure) makes the correction form, disclaimers, and ToS non-negotiable for launch. They are sequenced last because they depend on detail pages existing, but they must be gating requirements for v1 launch.

### Research Flags

Phases likely needing deeper research during planning:
- **Phase 1 (Data Foundation):** The ownership narrative content problem is not a code problem — it requires a content strategy decision. How to template narratives that are unique enough to avoid thin content penalties while being sustainable to write and update for 525 records needs concrete guidance before the phase begins.
- **Phase 5 (Trust + Legal):** The exact schema.org markup pattern for a directory page (vs. the business's own page) has nuance. Validate the `WebPage` + `ItemPage` + `about: LocalBusiness` pattern with Google's Rich Results Test tooling before committing to the template.

Phases with standard patterns (skip research-phase):
- **Phase 2 (Core Static Site):** Astro's `getStaticPaths()` + Content Collections is the canonical pattern for data-driven static sites. Well-documented in official Astro docs.
- **Phase 3 (Search System):** Pre-built Fuse.js index pattern is documented in Fuse.js official docs. The build-time → static JSON → browser fetch pattern is established.
- **Phase 4 (Filters):** Client-side filter state management with URL query parameters is a standard web pattern. No novel integration work.

### Gaps to Address

- **Ownership narrative content strategy:** Research identifies that 525 unique narratives are required and that formulaic descriptions will be penalized, but does not prescribe how to produce them efficiently. During Phase 1 planning, decide: write all 525 before launch, or launch with a subset of complete listings and expand? If templating, what data points create sufficient uniqueness?
- **Correction form tooling:** Research recommends a simple form (Formspree, Google Forms, or similar) but does not evaluate specific tools for the MA context. Evaluate spam protection and email notification setup during Phase 5 planning.
- **Owner group page scope:** The data pipeline produces ~200 owner group pages. Research does not define exactly how "owner" is defined when the same person operates under multiple LLCs or when a dispensary was acquired mid-year. This normalization rule needs a decision in Phase 1 before the Zod schema is finalized.
